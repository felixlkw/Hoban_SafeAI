# Writing Style Reference — 강욱님 보고 문체 (슬라이드 적용판)

사용자의 Report Writing Skill을 PPTX 문구 작성에 맞게 압축한 레퍼런스.
SKILL.md의 문체 원칙으로 부족할 때 이 문서를 참조한다.

## 1. 핵심 철학

보고받는 사람이 **바로 판단**할 수 있도록 쓴다.
문장을 예쁘게 다듬는 것이 아니라, 핵심 메시지·이슈·판단 포인트·실행 방향이 드러나도록 구조를 잡는다.

- 두괄식: 결론 먼저. 슬라이드에서는 거버닝 메시지가 결론을 선언
- 개조식: 문장보다 불릿, 설명보다 판단 포인트, 배경보다 의사결정 사항
- 단정적·전략적·과장 없는 자신감. 한국 기업 임원 보고에 맞는 절제된 표현

## 2. 거버닝 메시지 작성 기준

슬라이드 1장 = 판단 1개. 거버닝 메시지는 그 판단을 한 문장으로 선언한다.

좋은 예:
- "AI Factory는 단기 PoC 수행보다, 성공사례를 자산화하고 섹터·플랫폼으로 확산시키는 구조로 운영하는 것이 적절함."
- "현재 요청은 수행 가능하나, Factory가 지속 개발지원 조직처럼 운영되는 것은 지양 필요."
- "Q1에 운영체계를 공식화하고, Q2에 공동 오퍼링 4개를 출시함."

나쁜 예 (묘사형·번역투):
- "본 슬라이드에서는 Q1과 Q2의 주요 활동을 살펴봅니다."
- "다양한 이니셔티브를 통해 시너지를 창출할 수 있을 것으로 기대됩니다."

## 3. 번역투 변환표 (상세)

| 번역투 | 보고 문체 |
|---|---|
| ~를 수행할 수 있습니다 | ~ 가능 |
| ~하는 것이 필요합니다 | ~ 필요 |
| ~를 통해 …를 달성/실현합니다 | ~로 … 확보 / ~로 … 달성 |
| ~에 대한 검토가 요구됩니다 | ~ 검토 필요 |
| ~되어집니다, ~로 여겨집니다 | ~임, ~로 판단 |
| ~라고 할 수 있습니다 | ~임 |
| ~하는 것을 목표로 합니다 | 목표: ~ |
| ~함으로써 | ~로, ~해 |
| ~에 있어서 | ~에서, (삭제) |
| ~와 관련하여 | ~ 관련, (삭제) |
| ~의 경우 | ~는, (삭제) |
| 다양한 / 광범위한 / 포괄적인 | 구체 항목 나열로 대체 |
| 성공적으로, 효과적으로, 원활하게 | 삭제 또는 수치·기준으로 대체 |
| 강력한, 혁신적인, 최첨단의 | 삭제 |
| 시너지를 창출 | 구체 효과로 대체 (예: "제안 반영률 60% 이상") |
| ~할 예정입니다 | ~ 예정, ~함 |
| 또한, 그리고, 아울러 (문두 남발) | 삭제 — 불릿이 연결을 대신함 |

## 4. 선호 표현 (그대로 사용 가능)

- "~로 정리 가능"
- "~ 관점에서 접근 필요"
- "~ 구조로 가져가는 것이 적절"
- "~ 리스크 존재"
- "~ 기준으로 우선순위화 필요"
- "~까지는 Factory가 리드, 이후 확산은 담당 조직 이관"
- "단발성 수행보다 표준 자산화 관점에서 접근 필요"
- "의사결정 포인트는 다음과 같음"
- "핵심은 ~임" / "방향성은 ~로 가져가는 것이 적절함" / "단, ~ 리스크 존재"

## 5. 금지 표현

- "아마", "약간", "느낌", "일단 해보면"
- "좋을 것 같습니다" 반복
- "지원해드리겠습니다", "열심히 하겠습니다" 류 정성 중심 표현
- 장황한 배경 설명, 마케팅 수식어
- 가벼운 음슴체: "이건 좀 아닌 듯", "대충 이런 느낌임"

## 6. 관점 전환 원칙 — "지원"보다 "구조"

문구는 단순 지원 내역이 아니라 운영모델·책임구조·확장구조·자산화 구조를 드러낸다.

| 지양 | 지향 |
|---|---|
| 단발성 지원 | 표준 운영모델 |
| 개별 PoC | 재사용 가능한 자산 |
| 요청 대응 | 우선순위 기반 수요관리 |
| 개인 역량 | 조직 역량화 |
| 프로젝트 수행 | 사업화 플랫폼 |

예시 표현:
- "단순 지원이 아니라, PoC → 검증 → 표준 자산화 → 확산으로 이어지는 구조 필요"
- "초기에는 Factory가 품질을 보증하고, 이후 섹터·플랫폼이 확장 수행하는 구조가 바람직함"

## 7. 리스크·판단 포인트 포함

원문에 없어도 슬라이드 구성상 자연스러운 위치(콜아웃·하단 밴드)에 리스크와 판단 기준을 드러낸다.
단, 콘텐츠 원칙(원문 충실)과 충돌하지 않도록 사용자 원문에서 도출 가능한 범위로 한정한다.

예시:
- "단, Factory가 장기 개발조직으로 인식될 경우 본래 취지와 다르게 운영될 우려 있음"
- "따라서 PoC 범위와 이관 기준을 사전에 명확히 하는 것이 필요"

## 8. 보고 대상별 톤

| 대상 | 톤 |
|---|---|
| 부대표·파트너 보고 | 핵심만, 리스크·판단사항 중심, 음슴체, 불릿 |
| 고객 제안 | 신뢰감 있는 문장형, 구조·효과 중심, 내부 용어 제거 |
| 팀 내부 공유 | 실행 항목 중심, 담당자·일정·산출물 명확화 |

## 9. 문장 변환 예시

원문: "이건 우리가 할 수는 있을 것 같은데 계속 우리가 다 해주는 건 아닌 것 같고, 처음에 잘 보여주고 나머지는 담당 조직이 가져가는 게 맞을 것 같아요."
변환: "요청사항은 수행 가능하나, AI Factory가 지속 개발지원 조직처럼 운영되는 것은 취지와 다름. 초기 PoC를 통해 성공사례와 표준 자산을 확보한 후, 후속 확산은 담당 섹터·플랫폼 조직이 이어받는 구조가 적절함."

원문: "고객이 원하는 건 맞는데 기존 시스템을 그대로 두고 AI가 다 해달라는 느낌이라 좀 아쉬워요."
변환: "고객 요청은 PoC 관점에서 수행 가능하나, 기존 Legacy 구조를 유지한 상태에서 AI가 보완·대체하는 접근은 근본적 개선 측면에서 한계 존재. 단순 자동화보다 업무 프로세스 재설계 가능성까지 함께 제시하는 방향이 필요함."
