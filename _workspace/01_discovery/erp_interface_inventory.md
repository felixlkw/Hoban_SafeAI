# ERP I/F 인벤토리 & 보안 정책 요약 (Discovery)

- **문서 ID**: 01_discovery / erp_interface_inventory
- **작성**: erp-integration-engineer
- **작성일**: 2026-06-10
- **상태**: 가정 기반 초안 (ERP 운영팀 인터뷰 자료 부재)
- **전제**: 호반그룹 ERP가 PoC의 **유일한 외부 I/F**. 안전 DB 접근은 ERP를 경유하거나 ETL 동기화로만 가능.

> **중요 면책**: 본 문서의 모든 인터페이스/SLA/보안 수치는 ERP 운영팀 인터뷰 자료가 없는 상태에서 **국내 대형 건설그룹 ERP(SAP ECC/S4HANA 또는 자체 구축 J2EE/.NET 기반)의 일반 패턴**을 기준으로 한 합리적 가정이다. 운영 전환 전 실 ERP 검증이 필수이며, 모든 가정에는 `[검증 필요]` 태그를 붙였다. PoC 데모는 Mock 어댑터로 진행하므로 본 가정이 데모를 막지 않는다.

## 1. ERP I/F 인벤토리 (가정 기반)

각 인터페이스 옵션의 가용성·인증·SLA·rate limit·페이로드 제약을 정리한다. 가용 여부는 **가정 등급**으로 표기: ⭕(존재 가능성 높음) / △(조건부) / ❓(불확실) / ❌(권장 안 함).

| I/F 종류 | 가용 가정 | 인증 | SLA (가정) | Rate limit (가정) | 페이로드 제한 (가정) | 읽기/쓰기 | 비고 |
|---------|----------|------|-----------|------------------|---------------------|----------|------|
| **REST API** | △ ❓ | OAuth2 client_credentials 또는 JWT (API Gateway 발급) | p95 < 1.5s, 가용성 99.5% (평일 업무시간) | 분당 60~120 req `[검증 필요]` | 단건 ~1MB, JSON UTF-8 | 읽기+쓰기 | **최선호**. 신규 ERP는 REST 게이트웨이 보유 가능성. 안전모듈 전용 엔드포인트는 미존재 가능성 높음 → 신규 개발 필요 |
| **SOAP / WS** | ⭕ ❓ | WS-Security (X.509) 또는 Basic over TLS | p95 < 3s, 가용성 99% | 미명시(EAI 큐 기반) | 단건 ~5MB, XML | 읽기+쓰기 | SAP/레거시 ERP 표준. SAP의 경우 **RFC/BAPI를 SOAP로 노출**(PI/PO) 가능성 높음. verbose payload |
| **OData (SAP)** | △ ❓ | SAML/OAuth2 (SAP Gateway) | p95 < 2s | 분당 60 req `[검증 필요]` | 페이지네이션 강제(top/skip) | 읽기 위주, 쓰기 제한 | SAP S/4HANA면 OData v2/v4 우선. 마스터 조회에 적합 |
| **DB Direct (DB Link/JDBC)** | △ ❌ | DB 전용 계정(SELECT only) | - | - | - | 읽기 전용만 허용 | **쓰기 금지**. 읽기조차 ERP 팀이 통상 거부. read-replica/standby 제공 시에만 조건부 허용 |
| **File EDI (SFTP/배치)** | ⭕ | SFTP key + IP 화이트리스트 | 시간/일 단위 배치 | - | 파일 단위(수십~수백 MB) | 읽기+쓰기(배치) | 안전 DB 덤프 추출에 현실적. CSV/고정폭/XML. **ETL 동기화의 1순위 후보** |
| **Message Queue (MQ/Kafka)** | △ ❓ | mTLS 또는 SASL | 실시간 | 토픽 처리량 | 메시지 ~256KB~1MB | 읽기+쓰기(이벤트) | IBM MQ/Kafka 보유 시 등록 이벤트 송신에 이상적. PoC 단계 과투자 가능성 |

### 1.1 인터페이스별 의사결정 요지
- **JHA 등록(쓰기)**: REST(있으면) > SOAP/BAPI(SAP) > MQ 이벤트. 셋 다 동기/비동기 모두 Outbox 패턴으로 감싼다.
- **안전 DB 조회(읽기, RAG 학습용)**: File EDI 배치 추출 > OData 페이지네이션 조회 > (최후) DB read-replica.
- **마스터 코드 조회**: OData/REST 단건 조회 또는 EDI 마스터 덤프(일 1회).

### 1.2 PoC 데모 채택 가정
- PoC 데모는 **REST 계약(JSON)을 기준 인터페이스로 가정**하고 Mock 어댑터를 구현한다. 실 ERP가 SOAP/BAPI로 판명되면 `SoapErpAdapter`로 교체하되, 백엔드가 의존하는 `ErpAdapter` 추상 인터페이스는 불변(어댑터 패턴으로 격리).

## 2. 안전 DB 접근 전략 예비 비교

본 PoC가 안전 DB의 위험요인 데이터를 벡터 인덱스로 가져오는(읽기) 전략 비교. (쓰기=JHA 등록은 별도, 4장 register flow에서 다룸.)

| 옵션 | 설명 | 장점 | 단점 | ERP 부담 | 신선도 | PoC 적합 |
|------|------|------|------|---------|--------|---------|
| **A. ERP API 우회** | ERP가 제공하는 API로 안전 DB를 간접 조회 | 보안/권한 일관, ERP가 source of truth 보장 | 안전모듈 전용 API 미존재 가능성 → 신규 개발, ERP 부하 | 높음 | 실시간 | △ (API 존재 시) |
| **B. 전용 read-only API 신규** | 안전 DB 전용 read-only API를 ERP 팀이 신규 개발 | 관심사 분리, 성능 최적, 권한 명확 | ERP 팀 개발 일정 직접 영향, PoC 일정 리스크 | 중간(개발) | 실시간 | △ (일정 여유 시) |
| **C. ETL 배치 동기화** | 안전 DB → (EDI 덤프/OData 페이지 조회) → 벡터 인덱스 저장소로 주기 배치 | **PoC 분리·빠른 도입**, ERP 운영 부하 최소(야간 1회), 읽기 권한만 필요, 재인덱싱 제어 용이 | 신선도 지연(일 단위), ETL 파이프라인 구축 필요 | **낮음** | 일/시간 단위 | ⭕ **권장** |
| **D. CDC (Change Data Capture)** | DB 트랜잭션 로그 기반 변경분 실시간 캡처 | 신선도 최상, 부하 분산 | 로그 접근 권한·인프라(Debezium 등) 부담 큼, DB팀 협조 난이도 | 중간(로그) | 준실시간 | ❌ (PoC 과투자) |

### 2.1 PoC 권고: **옵션 C (ETL 배치 동기화)**

**근거:**
1. **ERP 일정 비의존** — ERP 팀의 신규 API 개발(B)이나 로그 접근(D) 없이, 읽기 권한 + EDI 덤프 또는 OData 조회만으로 착수 가능. PoC 일정 리스크 제거.
2. **ERP는 신성한 시스템** 원칙 준수 — 야간 배치 1회로 운영 부하 최소화. 온라인 트래픽 미발생.
3. **권한 분리 명확** — 읽기 전용 계정으로 안전 DB만 SELECT. 쓰기(JHA 등록)와 credential·화이트리스트 분리.
4. **신선도 충분** — 위험요인 마스터(4,469행)는 일 단위 갱신으로 충분. 분/초 단위 신선도 불요.
5. **재인덱싱 제어** — 배치 시점에 변경분 감지(hash diff) → 부분 재인덱싱 + 회귀 평가 트리거를 깔끔히 끼울 수 있음(data-engineer 협업).

**운영 전환 후 재검토:** 안전 데이터 실시간성 요구가 생기면 옵션 A(ERP API) 또는 D(CDC)로 점진 전환. 단 PoC 검증 완료 전에는 C 유지.

**검증 필요 항목:** EDI 추출 가능 여부 / OData 페이지네이션 한도 / 야간 배치 윈도우 시간대 / 안전 DB 읽기 계정 발급 가능 여부. 모두 `[검증 필요]`.

## 3. 마스터 데이터 식별

JHA 등록(쓰기) 및 RAG 조회(읽기)에 필요한 ERP 마스터를 식별하고 매핑 난이도를 예상한다.

> **핵심 발견**: 입력 Excel(`전사 하위공종 위험요인_20260518.xlsx`, 4,470행)은 분류를 **한글 명칭(대공종/중공종/세부항목)으로만** 보유하며 **코드 컬럼이 없다**. 즉 ERP 측 코드 체계와의 매핑이 **반드시 신규 구축**되어야 한다. 이것이 본 PoC의 가장 큰 마스터 데이터 리스크다.

| 마스터 | 본 PoC 측 형태 | ERP 측 가정 | 매핑 키 | 매핑 난이도 | 비고 |
|--------|---------------|------------|---------|------------|------|
| **공종 — 대공종** | 한글명 20종 (가설공사, 토목 전문공사 …) | ERP 공종/원가 코드 체계 (예: `HBC0XX`) `[검증 필요]` | 명칭 ↔ 코드 | **중** | 20종, 1:1 수작업 매핑 후 고정. 명칭 표기 차이(띄어쓰기/약어) 정규화 필요 |
| **공종 — 중공종** | 한글명 254종 | ERP 하위 공종 코드 `[검증 필요]` | 대공종 코드 + 중공종명 | **상** | 254종, 동의어/표기 흔들림(예: "타워크레인(T형)") 다수. 다대일/명칭불일치 위험 |
| **공종 — 세부항목** | 한글명 1,182종 | ERP에 동일 입도 존재 불확실 | 중공종 코드 + 세부명 | **상~최상** | ERP가 세부항목까지 코드화 안 했을 가능성. 그 경우 본 PoC 내부 코드 부여 + ERP 등록 시 중공종까지만 매핑 |
| **현장 (site_code)** | 등록 시 사용자 선택 | ERP 현장/프로젝트(WBS) 마스터 (예: `HB-SEOUL-001`) | site_code 직접 | **하** | ERP가 source of truth. 등록 페이로드에 코드 직접 전달. 조회 API 필요 |
| **작업자 (worker_id)** | 등록 시 사용자/승인자 | ERP 인사 마스터(사번, 예: `EMP-12345`) | worker_id 직접 | **하** | 코드 직접 사용. **PII 포함 → 외부 LLM 미전송**(data-engineer 화이트리스트와 정합 필수) |
| **부서 (dept_code)** | 등록 메타 | ERP 조직 마스터 | dept_code 직접 | **하** | 코드 직접 사용 |
| **재해형태** | 한글명 21종 (KRAS) | 표준 분류(KOSHA/KRAS) — ERP 코드 존재 불확실 | 명칭 ↔ 코드 | **중** | KRAS 표준이라 정형화 용이. ERP 미보유 시 본 PoC 표준 코드 사용 |

### 3.1 매핑 전략 요지 (상세는 03_design/erp_master_mapping.md에서 확정)
- **대공종(20)·재해형태(21)**: 소규모 → 수작업 1:1 매핑 룩업 테이블 우선 확정.
- **중공종(254)·세부항목(1,182)**: 자동 후보 매칭(명칭 유사도) + 안전/ERP 팀 검수. **미매핑은 `[미매핑]` 상태로 두고 검색에서 제외**(스킬 단계3 규칙).
- **충돌 시 ERP 코드 우선** (source of truth 원칙).
- 세부항목을 ERP가 코드화하지 않았다면, 본 PoC가 내부 코드를 부여하고 ERP 등록은 중공종 코드까지만 전송 + 세부는 텍스트 필드로.

## 4. 보안 요구 가정 (사내 보안 가이드라인 일반 패턴)

ERP 운영팀의 실제 가이드라인 부재 → 국내 대기업/건설그룹 일반 보안 정책 패턴 기반. 모든 항목 `[검증 필요]`.

### 4.1 네트워크 경계
- [ ] **전용 회선 또는 VPN** — 본 PoC 서버 ↔ ERP DMZ 간. 공중망 직접 노출 금지. `[검증 필요]`
- [ ] **IP 화이트리스트** — ERP 게이트웨이에 본 PoC 송신 IP 등록. **읽기용/쓰기용 IP 분리** 검토. `[검증 필요]`
- [ ] **통신 암호화** — 전 구간 TLS 1.2+ 강제. 가능 시 **mTLS**(상호 인증) 또는 OAuth2 client_credentials. `[검증 필요]`

### 4.2 인증·시크릿
- [ ] **시크릿 저장소** — Vault 또는 cloud Secrets Manager 사용. 환경변수/코드 평문 노출 금지. (PoC는 `.env`+gitignore, 운영은 Vault)
- [ ] **credential 회전** — ERP 계정/토큰 90일 회전 정책. `[검증 필요]`
- [ ] **권한 분리** — **읽기(ETL) 계정**(안전 DB SELECT only)과 **쓰기(JHA 등록) 계정**(등록 API 단일 권한) 물리적 분리. 서로의 권한 침범 불가.

### 4.3 데이터 보호 / 컴플라이언스
- [ ] **PII 외부 LLM 미전송** — 작업자 ID/이름 등 인사정보는 외부 LLM 프롬프트에 절대 미포함. data-engineer 화이트리스트 필터와 정합. (제약사항 명시)
- [ ] **외부 LLM 화이트리스트 필터 강제** — 전송 페이로드 사전 검사(제약사항).
- [ ] **감사 로그** — ERP 모든 호출(읽기/쓰기) 감사 로그 1년 이상 보관. 요청 주체·시각·idempotency_key·결과 기록. `[검증 필요]` (보관 기간)
- [ ] **최소 권한 원칙** — 안전 DB의 위험요인 테이블 외 타 ERP 테이블 접근 불가.

### 4.4 위변조 방지 (safety-domain-expert 수신 대기)
- [ ] JHA 등록 페이로드에 **승인자(approved_by)·승인시각(approved_at) 필수** + 무결성 보장(서명/해시 검토). safety-domain-expert의 법정 필수 필드 체크리스트 수신 후 확정.

## 6. PoC 데모 전략 (Mock 어댑터 범위)

PoC 데모는 실 ERP 없이 **`MockErpAdapter`로 전 시나리오를 결정적(deterministic)으로 재현**한다. 백엔드는 `ErpAdapter` 추상 인터페이스에만 의존하므로 어댑터 교체로 실 ERP 전환 가능.

### 6.1 Mock이 구현할 메서드 (스킬 단계8 기준)
- `register_jha(payload, idempotency_key) -> RegisterResult`
- `get_master_codes(kind) -> list[MasterCode]` (major/sub/detail/site/worker)
- `health_check() -> HealthStatus`

### 6.2 데모 시나리오 (결정적 트리거)
| 시나리오 | 트리거(Mock 입력) | 기대 동작 | 검증 포인트 |
|---------|------------------|----------|------------|
| **S1. 등록 성공** | 정상 payload | `erp_jha_id` 발급 + `REGISTERED` 반환 → 세션/프론트에 ID 표시 | 발급 ID 회수·표시 |
| **S2. 재시도 후 성공** | `idempotency_key`에 `flaky-` 접두 | 첫 N회 5xx/timeout → 지수 백오프 재시도 → 성공 | Outbox 재시도·"등록 대기" UX |
| **S3. Idempotency 충돌** | 동일 `idempotency_key` 재전송 | 409가 아닌 **200 + 기존 ID** 반환(정상) | 중복 등록 없음 |
| **S4. 영구 실패(마스터 미존재)** | `major_type_code`에 미매핑 코드 | `ErpFatal` → 등록 보류 + 사용자/운영팀 알림 + 매핑 갱신 요청 | 4xx 비재시도, 매핑 미스 카운트 |
| **S5. 인증 실패** | `idempotency_key`에 `expired-` 접두 | 401/403 → **즉시 중단**(재시도 금지) + 운영팀 알림 | 시크릿 만료 경로 |
| **S6. ERP 다운(장애 격리)** | health_check FAIL 모드 | Outbox 적재 + "등록 대기 중" 표시, **RAG 추천은 정상** | 경계 격리, queue depth 메트릭 |
| **S7. 마스터 조회** | `get_master_codes("major")` | 고정 20종 대공종 코드 반환(매핑 룩업 시드) | ETL/매핑 시드 데이터 |

### 6.3 Mock 데이터 시드
- 대공종 20 / 재해형태 21은 Excel 실 명칭에서 추출하여 `HBC0XX` 가상 코드를 부여한 고정 룩업으로 시드(03_design 매핑 CSV와 동일 소스).
- 현장/작업자/부서는 소량 가상 마스터(예: `HB-SEOUL-001`, `EMP-12345`, `DEPT-EHS`)로 시드.

### 6.4 운영 전환 경계
- Mock → (스테이징 ERP smoke test) → 실 ERP. **운영 전환 전 실 ERP 통합 테스트 필수**(질문 #1~#5, #8, #13 답변 확보가 전제).
